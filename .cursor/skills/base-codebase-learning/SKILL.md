# Base Codebase 学习课程规范

## 项目定位

本仓库是**多语言编程学习基地**，不是某个语言的完整项目。

- **学什么**：编程功能（变量、循环、函数、类、异步……）+ 各语言的表达方式
- **怎么学**：一课一课进阶，每课只讲一个知识块
- **核心信念**：复杂功能 = 简单功能的堆砌

当前已有：`dart_coding/`。未来可增 `python_coding/`、`javascript_coding/` 等，**各语言目录独立、规范统一**。

## 目录结构

```
base-codebase/
├── {lang}_coding/          # 一种语言一门课，如 dart_coding/
│   ├── 01_hello.{ext}
│   ├── ...
│   ├── README.md           # 该语言完整课表与运行说明（必须）
│   ├── lib/                # 仅某课需要时
│   └── test/               # 测试课
└── .cursor/skills/
```

每种语言一个文件夹，命名：`{语言}_coding`（小写 + 下划线）。

## 一课 = 一个知识块

| 原则 | 要求 |
|------|------|
| **简洁** | 能讲清就不用冗长；只留演示必需的代码 |
| **单一** | 一文件聚焦一个功能点；融合时用分段注释 |
| **明了** | 运行即有清晰输出；文件名见名知意 |
| **进阶** | 严格按序号从简单到难；前后课有依赖关系 |

## 文件命名

```
NN_topic.{ext}
```

- `NN`：两位序号，01 起，**难度递增**
- `topic`：英文功能名（如 `04_loops`），**不用纯数字**（❌ `1.dart`）
- 测试课：`test/NN_topic.{ext}`

## 融合规则（通用）

**可融合**：同一知识块、自然衔接的内容（如 变量+类型、for+while）

**保持独立**：
- 需要先学 A 才能学 B 的（如 先 list 再 for-in）
- 概念差异大的（如 if 与 switch、async 与 stream）
- 跨维度的（如 集合 vs 面向对象）

**禁止过分融合**：一个文件塞多个 unrelated 主题。

融合时在代码内用分段注释：

```
// --- 1. 小节名 ---
// --- 2. 小节名 ---
```

## 文件头模板

每课文件头部必须包含（注释语言与项目一致，当前用中文）：

```
// 第 NN 课 / 共 XX 课：本课标题
// 运行：<该语言的运行命令>
// 上一课：... | 下一课：...
//
// 本课要点（一两句）；融合课注明分几步递进
```

## 注释

- 中文，充足但不啰嗦
- 解释「做什么、为什么」，不是复述语法
- 关键行注释；融合课用 `---` 分段

## 新增 / 修改课程流程

1. 确定属于哪种语言目录（或新建 `{lang}_coding/`）
2. 确定课序号与在路线中的位置
3. 判断独立成课还是与相邻课融合
4. 编写示例：可运行、输出清晰、代码极简
5. 更新相邻课的上一课/下一课链接
6. 更新该语言目录内**所有文件**的「共 XX 课」
7. **同步更新 `{lang}_coding/README.md`**（课表、运行命令、学习顺序、融合说明）

## 边界

- 不实现用户未要求的功能；有歧义先提问
- 不增加无关实体、依赖、抽象
- 辅助文件只为某一课服务，保持最小
- 改课程必改 README

## 反模式

- ❌ 纯数字文件名
- ❌ 一文件多主题且无分段
- ❌ 跳过序号或难度乱序
- ❌ 示例过长、演示多个不相关场景
- ❌ 只改代码不同步 README
- ❌ 把某语言规范写死成全项目唯一标准（应可复用到新语言）

## 检查清单

- [ ] 文件在正确的 `{lang}_coding/` 目录
- [ ] 命名 `NN_topic.{ext}`，序号连续、难度递增
- [ ] 一文件一知识块，简洁可运行、输出清晰
- [ ] 文件头：课次、运行命令、上下课链接
- [ ] 融合课有分段注释
- [ ] 该语言 README 已同步

## 语言实例

各语言的具体课表、运行命令、分析工具见对应目录 README：

- Dart：`dart_coding/README.md`（当前 22 课）

新增语言时，复制上述结构模式，课次内容按该语言特性调整，**规范不变**。
